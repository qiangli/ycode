- **Directory Structure**: project layout at depth 2.
- **Code Architecture**: a knowledge graph analysis showing module communities (logical boundaries), god nodes (architectural linchpins with the most connections), and cross-community edges (coupling points). Synthesize these into architecture descriptions — don't list the raw data.

## What to extract

Look for the highest-signal facts for an agent working in this repo:
- exact developer commands, especially non-obvious ones — how to run a single test, a single package, a focused verification step
- required command order when it matters, such as `lint -> typecheck -> test`
- code style guidelines: imports, formatting, types, naming conventions, error handling patterns that differ from defaults
- monorepo or multi-package boundaries, ownership of major directories, and the real app/library entrypoints
- framework or toolchain quirks: generated code, migrations, codegen, build artifacts, special env loading, dev servers
- repo-specific style or workflow conventions that differ from defaults
- testing quirks: fixtures, integration test prerequisites, required services, flaky or expensive suites
- important constraints from existing instruction files worth preserving
- concrete guardrails: directories that must never be modified, commands that must never be run, patterns that are prohibited — use "NEVER", "MUST NOT", "ALWAYS" with specific subjects

### Architecture depth

Use the code architecture analysis (communities, god nodes) to identify:
- **Non-obvious internal packages**: an agent can read filenames but cannot infer that `internal/vncauth/` implements Apple Remote Desktop authentication. Describe what key internal packages actually do.
- **Entry points and call chains**: document the actual entry point (e.g., `cmd/ycode/main.go → cobra CLI → REPL`).
- **Build patterns that aren't obvious**: embedded assets, code generation, build tags, cross-compile steps.
- **Service topology**: for multi-service repos, include a service table with tech stack and port numbers.

### Operational knowledge

- **Environment files**: if `.env` files are machine-specific and gitignored, say so explicitly.
- **Script purposes**: summarize what key scripts do — focus on the non-obvious ones.
- **Data persistence**: warn about commands that destroy persistent data.

Good `AGENTS.md` content is usually hard-earned context that took reading multiple files to infer.

## Writing rules

Include only high-signal, repo-specific guidance such as:
- exact commands and shortcuts the agent would otherwise guess wrong
- architecture notes that are not obvious from filenames
- conventions that differ from language or framework defaults
- setup requirements, environment quirks, and operational gotchas
- references to existing instruction sources that matter

Prefer fenced code blocks with exact commands over prose descriptions.
The Makefile or build tool is the source of truth for build commands — reference it, don't paraphrase it.

### Interoperability

The output AGENTS.md must work for any AI coding tool (Claude Code, opencode, Cursor, Copilot). Use tool-agnostic language. If CLAUDE.md exists and contains useful tool-specific guidance, reference it:
`**Read [CLAUDE.md](./CLAUDE.md)** for Claude Code-specific conventions.`

### Delegation over duplication

Sub-directories may have their own `AGENTS.md` or `CLAUDE.md` files. When the root AGENTS.md touches a topic that a sub-directory file covers:
- **Point to the sub-directory file** instead of inlining the detail.
- The root AGENTS.md provides the high-level map; leaf files own the specifics.

### What to exclude

- generic software advice ("write clean code", "follow best practices")
- long tutorials or exhaustive file trees
- obvious language conventions
- speculative claims or anything you could not verify from the provided context
- symbol-level details that the code structure map already shows — agents have access to repo map and graph query tools
- directory listings — agents have directory listing tools; describe non-obvious structure only
- content that restates language or framework defaults
- **"Open Questions" sections** — if context is incomplete, the output should simply be shorter
- verbose tables — summarize or point to the authoritative source file instead

When in doubt, omit.

Prefer short sections and bullets. If the repo is simple, keep the file simple. If the repo is large, summarize the few structural facts that actually change how an agent should work.

If `AGENTS.md` already exists at `{{PATH}}`, improve it in place rather than rewriting blindly. Preserve verified useful guidance, delete fluff or stale claims, and reconcile it with the provided project context.
