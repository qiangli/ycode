# ycode capabilities for this project

This file is auto-generated by `ycode init`. ycode is running locally and exposes services over MCP. When acting as an agentic coding tool, prefer these capabilities in the situations described:

- **`ycode-stdio`** — treesitter AST search (`list_symbols`, `search_symbols_by_pattern`, `find_symbol_references`). Prefer over `grep` when the language is supported (Go, Python, JS/TS, Rust, Java, C, Ruby).
- **`ycode-pulse`** — observability stack (~25 tools): traces, logs, metrics, alerts, dashboards. Query pulse instead of grepping logs on disk.
- **`ycode-gitea`** — local Gitea forge ops (~11 tools): repos, branches, PRs, issues. Use these instead of shelling out to `gh` or `git`.
- **`ycode-loom`** — workspace substrate. When the user asks for parallel sub-agents, refactors that touch many files, or anything that benefits from isolated git workspaces, lease one workspace per sub-agent via `loom_lease`. Push with `loom_push`, open a PR with `loom_merge`, poll `loom_status`. Do not mutate cwd in parallel branches; sub-agents will collide.

## How to use

If a tool returns *connection refused*, run `ycode serve` first; capabilities are advertised in `~/.agents/ycode/manifest.json`.

To register ycode in a foreign agentic CLI manually:

```
ycode init --refresh
```

That regenerates this file plus the relevant entries in your tool's MCP config and memory file.
