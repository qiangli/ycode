# Memory Management in Aider (x/aider/)

This document summarizes how Aider (the Python CLI coding agent) handles multi-turn conversations, context engineering, harness engineering, and memory management.

---

## 1. Architecture Overview

Aider is a CLI pair-programming agent built in Python. Its defining feature is the **repository map** -- a tree-sitter AST-based skeleton of the codebase that provides structural awareness without consuming full-file tokens. It uses git as a memory/undo mechanism.

---

## 2. Repository Map (Structural Memory)

### How the Map is Built

File: `aider/repomap.py` (927 lines)

Aider uses **tree-sitter** via the `grep_ast` library to parse source files into ASTs and extract definitions/references:

1. Language-specific `.scm` (Scheme) query files loaded for tree-sitter
2. Files parsed into AST trees
3. Queries extract `name.definition` and `name.reference` tags
4. Results stored as named tuples: `(rel_fname, fname, line, name, kind)`

### What's Included

- Classes, functions, methods, signatures (via tree-sitter tags)
- Definitions (`kind="def"`) and references (`kind="ref"`)
- Binary search ranks tags by relevance to find optimal map size

### Token Budgeting

| Scenario | Budget |
|----------|--------|
| Files in chat | `max_map_tokens` (default 1024) |
| No files in chat | `max_map_tokens * 8` (default 8192) |
| Upper bound | `min(multiplied, max_context_window - 4096)` |
| Binary search tolerance | 15% of target budget |

### Refresh Strategy

| Mode | Behavior |
|------|----------|
| `auto` | Cache if `map_processing_time > 1.0s` |
| `files` | Refresh when file mtimes change |
| `manual` | Only on explicit request |
| `always` | Never cache |

Cache: SQLite-backed `diskcache.Cache` in `.aider.tags.cache.v3` directory. Invalidated by file mtime changes.

## 3. Multi-Turn Conversation

### Two-Level Message Storage

File: `aider/coders/base_coder.py`

- `cur_messages`: Current turn (user + assistant + reflection loops)
- `done_messages`: Previous turns (accumulates across conversation)

After each turn: `cur_messages` → appended to `done_messages`, then reset.

### Turn Flow

1. `run_one()` — processes user input
2. `send_message()` — sends to LLM with full context
3. If lint errors or file mentions → set `reflected_message` and loop
4. Max reflections limit prevents infinite loops
5. Shell output integrated via user/assistant message pairs

### Auto-Summarization

File: `aider/history.py`

When `done_messages` grows too large:
- `ChatSummary` spawns async summarization thread
- Uses weak model (cheaper/faster) for summarization
- Budget: `min(max(max_input_tokens/16, 1024), 8192)` tokens
- Recursive strategy: splits at 50% of budget, summarizes older half
- Depth limit: 3 iterations before summarizing everything

## 4. System Prompt / Context Engineering

### Message Order (Context Window Layout)

File: `aider/coders/base_coder.py`, `ChatChunks` class

```
┌─────────────────────────────────────────────┐
│  system        System prompt + rules         │  Most cacheable
│  examples      Few-shot examples (optional)  │
│  readonly_files  Reference files (full)      │
│  repo          Repository map (AST skeleton) │
│  done          Previous turns (summarized)   │
│  chat_files    Editable files (full content) │
│  cur           Current turn messages         │
│  reminder      System reminder (rules recap) │  Most recent
└─────────────────────────────────────────────┘
```

### File Context Strategy

| Category | Content | Budget |
|----------|---------|--------|
| Chat files | Full file content (editable) | Uncapped |
| Repo map | AST skeleton (function/class signatures) | 1K-8K tokens |
| Read-only files | Full content (non-editable) | Uncapped |
| Images | Base64 with vision models | Model-dependent |

### Model-Specific Adaptations

- `system_prompt_prefix` — model-specific prefix
- `lazy` prompt — "You are diligent and tireless!" for lazy models
- `overeager` prompt — "Pay careful attention to scope" for overactive models
- Fence format: quadruple backticks if model confuses triple backticks
- `reminder` placement: either as system message or stuffed into last user message

## 5. Git Integration as Memory

### Auto-Commit

After each successful edit:
1. All edited files auto-committed
2. Commit message generated by weak model
3. Uses conventional commit format (fix, feat, docs, etc.)
4. Co-author attribution via `--attribute-author` / `--attribute-committer`

### Undo/Revert

- `/undo` command → `git reset --hard HEAD~1`
- All aider commits tracked in `aider_commit_hashes` set
- `last_aider_commit_hash` / `last_aider_commit_message` stored

### Dirty File Handling

- Checks for uncommitted changes before editing
- Offers to commit dirty files first
- Shows diffs after auto-commit if `show_diffs=True`

## 6. Token Management

### Token Counting

File: `aider/models.py`

- Uses `litellm.token_counter()` for message lists
- Uses `litellm.encode()` for string tokenization
- Sampling-based estimation for large texts (> 200 chars): samples every Nth line

### Context Window Budget

| Component | Budget Source |
|-----------|-------------|
| Chat history | `min(max(max_input_tokens/16, 1024), 8192)` |
| Repo map | `min(max_input_tokens/8, 4096)` |
| System prompt | ~500-1000 tokens (varies by model) |
| Current turn | Remaining tokens |

### Context Overflow Handling

On `ContextWindowExceededError`:
1. Marks turn as `exhausted`
2. Shows error: suggests `/drop`, `/clear`, split files smaller
3. Increments `num_exhausted_context_windows` counter
4. Does NOT auto-compact — user must manually reduce context

## 7. Edit-Lint-Fix Cycle

### Reflection Loop

File: `aider/linter.py`

After edits are applied:
1. **Auto-lint** — runs linting on changed files
2. If errors found → asks user to fix
3. If confirmed → lint errors fed back as `reflected_message`
4. Model attempts fix in next reflection iteration
5. Max reflections limit prevents infinite loops

### Multi-Stage Python Linting

1. `basic_lint()` — tree-sitter syntax check (ERROR nodes)
2. `lint_python_compile()` — Python `compile()` for syntax errors
3. `flake8_lint()` — Flake8 style checking

### Auto-Test

After edits, if `auto_test=True`:
- Runs test command
- Test output can be fed back as reflected message for fix attempt

## 8. Prompt Cache Warming

File: `aider/coders/base_coder.py`

For Anthropic/Claude models with cache control:
- Background thread pings API every ~5 minutes
- Keeps prompt cache alive between user messages
- Cache headers: `cache_control: {"type": "ephemeral"}`
- Applied to last message of: examples, system/repo, chat_files
- Tracks: `prompt_cache_hit_tokens`, `cache_creation_input_tokens`

## 10. Comparison with ycode

| Feature | Aider | ycode |
|---------|-------|-------|
| **Structural memory** | Tree-sitter repo map | None |
| **File context** | Full files + AST skeleton | Full files only |
| **Compaction** | LLM summarization (weak model) | Heuristic 7-field summary |
| **History storage** | Markdown (append-only) | JSONL |
| **Git integration** | Auto-commit + undo | Git context only |
| **Token counting** | litellm (exact) | Heuristic (len/4+1) |
| **Lint cycle** | Auto-lint → reflect → fix | None |
| **Cache optimization** | Background warmup thread | Fingerprint-based TTL |
| **Context layout** | 8-tier hierarchical chunks | 2-tier static/dynamic |

### Key Features ycode Could Adopt

1. **Repository map** — tree-sitter AST skeleton for structural awareness (biggest gap)
2. **LLM-based summarization** — use model for higher-quality compaction summaries
3. **Edit-lint-fix reflection loop** — auto-lint after edits and retry on errors
4. **Hierarchical context layout** — prioritized message ordering with explicit tiers
5. **Dual-model strategy** — use cheaper model for summarization and commit messages
6. **Cache warmup** — background thread to keep prompt cache alive
7. **Git auto-commit with undo** — atomic commits per edit with `/undo`
